# TraeCN-Framework 架构说明

## 架构目标

TraeCN-Framework 的目标不是堆更多提示词，而是把 Trae 的工作方式拆成可维护的几层，让主 Agent、规则、技能、记忆和项目资产各司其职。

当前架构遵循两个原则：

1. **真实可执行**：文档、目录、规则必须和实际文件一致。
2. **职责清晰**：会话生命周期由 Coordinator 负责，专业任务由 Skill 负责。

## 五层架构

### 1. Coordinator 层

主 Agent 负责：

- 接收用户输入
- 判断是生命周期指令还是专业任务
- 维护 `TodoWrite` 状态
- 调度 Skill 或直接执行内建流程
- **主动式引导**：根据 Memory 状态主动推送复盘、分析或设计建议

这一层只做编排，不和具体领域逻辑耦合。

### 2. Rule 层

位于 `.trae/rules/`，负责定义全局约束：

- 身份与行为边界
- 记忆加载与更新规则
- 技能调度规则
- 错误处理与降级
- 会话管理与自动保存
- 输出规范

Rule 层回答的是“必须怎么做”。

### 3. Skill 层

位于 `.trae/skills/`，负责专业任务执行。当前内置 6 个技能：

- `分析`
- `设计`
- `测试`
- `部署`
- `复盘`
- `项目归档`

Skill 层回答的是“具体怎么做”。

### 4. Memory 层

位于 `Core/记忆/`，使用结构化 Markdown 表格存储本地记忆：

- `user_scope.md`：用户偏好与长期工作习惯
- `project_scope.md`：项目状态与阶段成果
- `session-handoff.md`：会话交接与恢复
- `knowledge/`：错误、模式、决策沉淀

Memory 层回答的是“已经发生了什么，之后如何继续”。

### 5. Workspace 层

位于项目根目录，承载运行期资产与自我进化：

- `Builds/`：已完成项目归档。
- `Skills/`：项目自定义技能扩展，支持动态发现。
- `Modules/`：**可选资产沉淀区**，只在出现真实高复用代码时启用。
- `Evolve/`：**框架进化中心**，记录架构演进（进化日志）与踩坑预防（错题本）。
- `Backup/`：关键记忆的离线备份与重置快照。

## 指令分流模型

为避免“所有事情都做成 Skill”导致的失真，框架将指令分成两类。

### A. Coordinator 内建指令

这类指令直接由主 Agent 执行，不强制映射为 Skill：

- `/初始化`
- `/重置`
- `/结束`

原因是它们本质上属于会话生命周期和框架维护，不是独立的专业领域任务。

其中 `/重置` 采用安全优先策略：

- 必须先说明影响范围并获得二次确认
- 必须先备份 `Core/记忆/` 到 `Backup/reset-时间戳/`
- 默认只重置记忆、知识库和会话状态
- 不删除 `.trae/`、`docs/`、`Builds/`、`Skills/` 等框架和项目资产
- 完成后回到可重新 `/初始化` 的干净状态

### B. Skill 指令

这类指令由主 Agent 调度到技能文档执行：

- `/分析`
- `/设计`
- `/视觉设计`
- `/测试`
- `/部署`
- `/复盘`
- `/项目归档`

## 数据流

```text
用户请求
  -> Coordinator 判断意图
  -> 选择执行路径
     -> 内建指令：直接执行
     -> 专业任务：调用 Skill
  -> 使用工具完成操作
  -> 更新记忆文件
  -> 返回结果给用户
```

## 执行闭环

从当前阶段开始，框架不再把“指令可触发”视为“能力已完成”。

所有内建流程与 Skill 都必须满足：

- 有预检：先读取配置、记忆和相关上下文
- 有执行：必须真的调用工具完成动作
- 有诊断：修改后做 `GetDiagnostics` 或错误溯源
- 有验证：重新执行关键命令或读取目标文件确认结果
- 有沉淀：把结果写回 Memory / Evolve / 文档
- 有回滚：高风险流程必须定义失败恢复路径

统一标准见 `.trae/documents/闭环执行矩阵.md`。

## 目录设计原则

### 目录必须和文档一致

README、规则文件和实际目录结构必须保持同步，避免出现“文档里有，项目里没有”的伪架构。

### 技能数量必须和规则一致

只有真正存在的技能目录，才能出现在技能调度表中。生命周期指令不再伪装成技能。

### 架构文档必须只有一套口径

不能同时存在“用 Markdown 记忆”和“严格禁用 Markdown 记忆”两套互相冲突的说法。

## 当前优化结果

本轮优化后，框架完成以下统一：

- 将架构口径统一为本地结构化记忆模型
- 将技能模型统一为 7 个真实技能
- 将 `/初始化`、`/重置`、`/结束` 归为 Coordinator 内建流程
- 补齐 `.trae/documents/` 作为内部设计文档目录
- 为 `/重置` 增加二次确认、备份与回滚约束
- **强化执行深度**：优化 01、03、06 号规则，明确跨项目进化、工具失效应对及输出链接规范

## 后续演进方向

- 补齐 `/初始化`、`/重置`、`/结束` 的验收、回滚与交付口径
- 提升 `分析`、`设计`、`测试`、`部署`、`复盘`、`项目归档` 的执行深度
- 为 `Evolve/` 定义固定沉淀格式
- 仅在出现真实复用代码后再扩展 `Modules/`
