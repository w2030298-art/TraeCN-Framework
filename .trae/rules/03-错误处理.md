# 错误处理

## 核心机制：根因自愈 (Root Cause Self-Healing)
任何错误处理必须遵循以下「闭环修复」流程：
1. **报错捕获**：立即通过 `CheckCommandStatus` 或终端输出获取原始错误。
2. **根因溯源**：调用 `GetDiagnostics` 或 `Grep` 搜索报错关键词，识别是语法错误、环境缺失还是逻辑冲突。
3. **策略降级**：若首选工具失败（如 `npm run dev` 报端口占用），自动尝试备选方案（如 `npm run build` 验证或更换端口）。
4. **验证交付**：修复后必须重新执行原命令，确认错误消失。

## 错误分级
### Level 1：工具/环境调用失败
- 分析原因：文件不存在/权限问题/参数错误/依赖缺失。
- 自愈策略：
  - `Read` 失败 -> `Glob` 查找。
  - `Vite` 未找到 -> 主动执行 `npm install`。
  - `Port` 占用 -> 建议修改 `server.port`。
- 记录错误：写入 `knowledge/errors.md`。

### Level 2：逻辑/构建执行失败
- 主 Agent 必须在交付前尝试 **至少 3 次** 不同维度的修复（语法 -> 配置 -> 依赖）。
- 若 3 次失败，记录失败路径至 `Evolve/错题本.md` 并转 Level 3。

### Level 3：需求理解失败
- 明确告知用户，提供选项
- 不猜测用户意图，但提供专业建议

### Level 4：系统级故障
- 快速写入 session-handoff.md 保存状态
- 告知用户建议开启新会话自动恢复

## 工具与性能限制处理
- **工具调用超时**：如果 `RunCommand` 或 `SearchCodebase` 超时，尝试拆分任务范围（例如分文件夹搜索）并重试一次；若仍失败，降级为 `Grep` 或询问用户。
- **上下文超长 (Token Limit)**：当检测到对话历史过长导致响应变慢或截断时，主动建议用户先结束会话（系统命令：`/结束`）以保存当前状态并开启新会话。
- **文件读取限制**：单次读取文件超过 2000 行失败时，必须使用 `offset` 分段读取。

## 重置失败处理

- `/重置` 过程中如果清理失败，优先从 `Backup/reset-时间戳/` 回滚 `Core/记忆/`
- 回滚后记录失败场景到 `knowledge/errors.md`
- 明确告知用户：哪些文件已恢复、哪些步骤未完成、是否需要再次尝试

## 黄金法则
永远不要把原始报错直接甩给用户，应提供：错误说明 + 可能原因 + 建议操作 + 询问用户选择

## 错误归档
每次解决后写入 `Core/记忆/knowledge/errors.md`：
- 场景、错误信息、原因分析、解决方案、预防措施
