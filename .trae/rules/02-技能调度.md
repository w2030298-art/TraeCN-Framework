# 技能调度

## 调度原则
1. **意图优先**：用户明确指定 > 规则自动触发 > 意图推断
2. **按需加载**：根据用户意图选择最相关的技能
3. **职责分离**：生命周期指令由 Coordinator 执行，专业任务才交给 Skill
4. **串行编排**：技能 A 输出 → 主Agent处理 → 技能 B 输入
5. **透明告知**：让用户知道正在调用哪个技能做什么

## 技能边界
- **全局初始化 Skill**：`traecn-init` Skill（安装在 `$HOME/.trae-cn/skills/traecn-init/`）在所有工作区中全局可用。当工作区不存在 `.trae/` 时，Coordinator 执行 `/初始化` 应调用该 Skill 完成 `.trae/` 的自动部署，再执行标准初始化。参见 `.trae/documents/初始化方式手册.md`。
- **全局钩子 (Global Hooks)**：Coordinator 在执行任何写操作或命令前，必须先读取 `.trae/config.md`。如果 `Global Hooks` 开关为 `ON`，则必须静默执行对应的预处理逻辑。
- **自愈机制 (Auto-Healing)**：当系统检测到代码存在 Linter 错误或命令执行失败时，Coordinator 必须跳过用户确认，主动尝试至少 2 次修复尝试。
- **上下文预测 (Cascade Mode)**：Coordinator 应通过 `LS` 或 `Glob` 实时同步当前编辑文件的周边目录结构，减少用户手动提供路径的负担。
- **自定义技能扩展**：用户可在 `Skills/` 目录下通过子目录方式新增技能，必须遵循 `Skills/SKILL_TEMPLATE.md` 模板。
- **动态发现**：Coordinator 在启动或识别到新意图时，会扫描 `Skills/` 目录下的 `SKILL.md` 文件。
- **统一执行合同**：所有内建指令与 Skill 都必须遵循 `.trae/documents/闭环执行矩阵.md` 中的“预检 -> 执行 -> 诊断 -> 验证 -> 沉淀”五段式闭环。

## 内建指令约束

- `/重置` 默认只清理 `Core/记忆/` 下的用户数据和会话状态，不删除框架代码与项目资产
- 执行 `/重置` 前必须先说明影响范围，并获得用户明确确认
- 执行 `/重置` 前必须先备份当前记忆到 `Backup/`
- 用户说“先暂停”“下次继续”时，不新增指令，统一归并到轻量结束会话流程（系统命令：`/结束`）

## ⚠️ 执行原则（核心约束）
**触发即执行**：当 `/指令` 触发技能时，必须：
1. **读取技能文档**：理解需要执行的步骤
2. **实际执行所有步骤**：使用 Read/Write/Delete/RunCommand 等工具
3. **禁止只输出文档**：不得仅输出技能文档内容而不执行实际操作
4. **验证执行结果**：确认文件/状态已按预期更新
5. **回复执行结果**：告知用户完成了什么操作

**反模式（绝对禁止）**：
- ❌ 触发技能后，只把技能文档输出给用户
- ❌ 描述"我会做什么"而不实际做
- ❌ 输出"执行完成"但没有实际写文件

**正模式（必须遵循）**：
- ✅ 触发技能后，读取文档 → 执行步骤 → 验证结果 → 回复确认

## 交付格式约束

Skill 或内建指令执行结束时，回复中至少应包含：
- **已完成动作**：实际做了哪些文件、命令或诊断操作
- **验证结果**：如何确认结果有效，或当前还缺哪一步验证
- **沉淀结果**：更新了哪些记忆、文档或经验记录
- **下一步建议**：用户此刻最值得继续做的 1-3 步

## 编排模式
- **线性**：单技能完成（如"帮我修复bug"→测试）
- **链式**：多技能串行（分析→设计→开发→测试）
- **分支**：根据问题类型选择技能
- **循环**：多次迭代（重构模块→分析→开发→测试→循环）

## 冲突处理
多个技能匹配时，询问用户优先做哪个

## 视觉类分流建议
- 用户提到海报、封面、配图、PPT、视觉风格、配色、排版、UI 界面、原型、美术优化、设计评审时，优先路由到 `视觉设计`。
- 用户提到技术选型、系统架构、模块拆分、接口边界、数据流时，优先路由到 `设计`。
- 用户只说“帮我设计一下”且意图不明确时，先询问是“系统设计”还是“视觉设计”。
- 若用户未显式输入 `/视觉设计`，但命中上述视觉类关键词，Coordinator 应直接切换到 `视觉设计` 流程，或先明确告知用户将按视觉设计处理。
