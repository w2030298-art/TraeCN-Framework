# 视觉设计工作流

这份文档用于补充 `SKILL.md` 中较短的执行合同，把 `视觉设计` 的完整方法论拆成可复用流程。

## 目的

- 让 `视觉设计` 不只是输出风格词，而是形成完整的分析、发散、收束、评审闭环。
- 把 huashu-design 中可迁移的阶段思路沉淀为当前框架可执行的版本。
- 在不让 `SKILL.md` 过重的前提下，保留完整的方法路径。

## 七阶段流程

### Phase 1. 设计哲学选型
- 目标：先决定这次设计以什么审美哲学为核心，而不是直接做版式。
- 关键动作：
  - 判断项目基调、用户气质、内容属性
  - 从 `references/design-styles.md` 中初选方向
  - 明确哪些气质是本次要强化的，哪些是要避免的
- 产物：
  - 方向候选
  - 风格关键词
  - 禁区列表

### Phase 2. 场景与载体建模
- 目标：明确设计发生在什么媒介、什么阅读距离、什么使用路径中。
- 关键动作：
  - 识别是封面、海报、PPT、官网、App UI 还是设计评审
  - 从 `references/scene-templates.md` 提取规格、重点和信息密度要求
  - 判断用户最在意的是传播力、专业感、可读性还是转化
- 产物：
  - 输出规格
  - 场景约束
  - 优先级排序

### Phase 3. 方向发散
- 目标：给出 3 个真正可比较的方向，而不是 3 个同义词。
- 关键动作：
  - 至少覆盖不同流派
  - 每个方向都写清适配原因、特征和风险
  - 避免 3 个方向都只是“极简风”的不同说法
- 产物：
  - 3 个差异化方向
  - 每个方向的适用条件
  - 推荐优先级

### Phase 4. 方案收束
- 目标：从多个方向中选出最适合当前任务的执行方案。
- 关键动作：
  - 根据用户反馈或上下文选主方向
  - 允许混合不同方向的局部特征，但必须说明为什么混合
  - 明确“保留什么、删掉什么、不要做什么”
- 产物：
  - 主方向
  - 混合策略
  - 取舍依据

### Phase 5. 设计产物生成
- 目标：输出可直接指导制作或生成的结果。
- 关键动作：
  - 生成配色、排版、层级、留白、构图建议
  - 需要时输出 AI 提示词、草图说明、页面结构建议
  - 对不同载体给出不同粒度的说明
- 产物：
  - 视觉方案
  - 配色与版式规范
  - 提示词或结构草图

### Phase 6. 五维设计评审
- 目标：在交付前做结构化评审，而不是靠直觉说“还不错”。
- 关键动作：
  - 使用 `references/critique-guide.md`
  - 评分维度固定为：哲学一致性、视觉层级、细节执行、功能性、创新性
  - 输出 `Keep / Fix / Quick Wins`
- 产物：
  - 评审报告
  - 问题清单
  - 快速修复项

### Phase 7. 交付与沉淀
- 目标：让一次视觉方案变成可以复用的资产，而不是一次性回答。
- 关键动作：
  - 回读是否覆盖目标、场景、规格、限制
  - 若形成可复用模板，写入 reference 或记忆
  - 明确下一步建议：继续深化、生成图稿、进入评审或实现
- 产物：
  - 最终交付摘要
  - 残留风险
  - 下一步建议

## 决策原则

- 先场景，后风格：没有载体和目标时，不先谈审美。
- 先比较，后推荐：至少给用户一次可比较的方向选择。
- 先收束，后细化：没定主方向前，不急着展开细节规范。
- 先评审，后交付：任何方案都要过一次结构化评审。

## 何时需要追问

- 不知道目标受众是谁
- 不知道输出载体是什么
- 不知道核心信息是“传播”“说明”还是“转化”

满足以上任一项时，优先追问，不要直接编造方案。

## 与 `SKILL.md` 的关系

- `SKILL.md`：负责触发、职责、步骤、验收，是执行合同。
- `workflow.md`：负责方法论展开，是流程手册。
- 两者配合使用，避免主技能文件越来越重。
