# 视觉设计技能

## ⚠️ 执行声明
本技能被触发时，**必须实际执行**视觉设计相关流程，
**不得仅输出空泛风格描述而不产出可执行方案**。

## 闭环要求
- 默认遵循 `.trae/documents/闭环执行矩阵.md` 的五段式闭环：预检 -> 执行 -> 诊断 -> 验证 -> 沉淀。
- 如果输出了视觉方案、提示词、页面草图或评审报告，必须说明验证范围与残留风险。

## 触发条件
- 用户输入 `/视觉设计` 时。
- 用户明确需要海报、封面、配图、PPT、视觉风格、配色、排版、UI 界面或原型时。
- 用户需要对现有视觉稿进行设计评审、方向选择或美术优化时。

## 职责
- 视觉需求澄清与设计方向建议
- 配色、排版、留白、层级、氛围与风格方案设计
- 封面、配图、信息图、PPT、落地页、App UI 的视觉思路输出
- 设计提示词、草图说明、结构化评审报告生成

## 前置要求
- 先读取当前项目结构、已有记忆和相关文档，避免重复给出与项目定位冲突的方案。
- 如需求模糊，最多补问 3 个关键问题：目标受众、核心信息、输出格式。
- 先读取 `workflow.md`，明确当前任务所处阶段，不要跳过“方向发散 -> 方案收束 -> 设计评审”。
- 需要根据场景主动读取以下参考文件：
  - `references/design-styles.md`
  - `references/scene-templates.md`
  - `references/critique-guide.md`
  - `examples.md`
  - `output-template.md`

## 执行步骤
1. **识别设计场景**
   - 读取当前项目上下文、用户要求和已有素材，判断属于封面、配图、PPT、信息图、官网还是 App UI。
   - 明确输出格式、目标受众、核心信息、情感基调和约束条件。
   - 需要先对照 `workflow.md` 中的 Phase 1 和 Phase 2，先定设计哲学与场景约束。
2. **顾问式重述需求**
   - 用 100-200 字重述需求本质、使用场景和设计目标。
   - 说明本次设计重点解决什么，不解决什么，避免过度发散。
3. **给出 3 个差异化方向**
   - 至少提供 3 个来自不同流派的视觉方向，方向名称必须包含设计师、机构或明确流派锚点。
   - 每个方向至少包含：为什么适合、核心特征、气质关键词、建议使用场景。
   - 禁止提供 3 个相近方向导致用户无法比较。
   - 需要先读取 `references/design-styles.md`，优先从不同流派中取方向。
4. **生成可执行设计产物**
   - 根据用户当前阶段，输出其一或组合：
   - 结构化视觉方案
   - 配色与版式规范
   - AI 生成提示词
   - 页面草图说明或 Demo 结构建议
   - 若用户已有偏好，允许混合不同方向的元素并解释取舍依据。
   - 需要结合 `references/scene-templates.md` 中的规格、场景重点和信息密度要求。
   - 如用户没有明确要求输出格式，可参考 `examples.md` 的输出骨架组织答案。
   - 优先使用 `output-template.md` 的标准模板组织结果；只在需求很轻时降级为简化模板。
5. **执行设计评审**
   - 对最终方案按以下 5 个维度打分：哲学一致性、视觉层级、细节执行、功能性、创新性。
   - 输出 `Keep / Fix / Quick Wins`，问题说明必须包含原因和可执行修改建议。
   - 需要参考 `references/critique-guide.md`，避免只给主观感受。
6. **验证与沉淀**
   - 回读输出内容，检查是否覆盖了目标、场景、规格与关键限制。
   - 若形成了可复用设计模式或关键决策，补充到记忆或文档中。

## 输出要求
- **需求重述**：我理解你要做什么，服务谁，用在什么场景。
- **场景与约束**：载体、规格、核心信息、情感基调与关键限制。
- **设计方向**：3 个有明显差异的方向及推荐理由。
- **推荐方向**：明确推荐项及混合取舍依据。
- **执行产物**：配色、版式、提示词、草图说明或 Demo 建议。
- **设计评审**：5 维评分、优点、问题和快速修复清单。
- **验证范围与风险**：说明已验证内容、未验证内容与残留风险。
- **下一步建议**：最适合继续深化或落地的 1-3 步。

## 验收标准
- 结果必须贴合当前项目上下文，而不是通用审美套话。
- 必须给出可执行方向，不允许只说“高级感、科技感、极简风”这类空词。
- 至少完成一次对输出结果的自检或评审。
- 若未直接产出图稿，也要产出足以指导下一步生成或落地的结构化说明。

## 异常处理
- 如果需求信息不足，先缩小问题范围，只追问最影响方向判断的 1-3 个问题。
- 如果用户对 3 个方向都不满意，回到方向阶段重新组合，而不是硬推原方案。
- 如果无法直接出视觉成品，降级为可执行提示词、版式规范和评审清单。
