# 内部架构设计

## 设计结论

TraeCN-Framework 采用“Coordinator + Rules + Skills + Memory + Workspace”的五层结构。

## 层级职责

| 层级 | 目录 | 职责 |
| --- | --- | --- |
| Coordinator | 主 Agent | 解析意图、维护进度、选择执行路径、主动式建议 |
| Rules | `.trae/rules/` | 约束行为、保存策略、错误处理、输出规范 |
| Skills | `.trae/skills/` | 承接分析、设计、测试、部署、复盘、归档等专业任务 |
| Memory | `Core/记忆/` | 保存用户偏好、项目状态、交接信息和知识沉淀 |
| Workspace | 根目录业务区 | 承载归档、扩展技能、进化记录（进化日志/错题本）、备份和可选资产沉淀 |

## 指令模型

### Coordinator 内建指令

- `/初始化`
- `/重置`
- 结束会话（系统命令：`/结束`）

### Skill 指令

- `/分析`
- `/设计`
- `/视觉设计`
- `/测试`
- `/部署`
- `/复盘`
- `/项目归档`

## 关键约束

1. 只有真实存在的技能才能出现在调度表中。
2. 生命周期操作不伪装成技能。
3. 规则、README、目录结构必须同步。
4. 记忆统一采用本地结构化文件，不再保留冲突说法。
5. **执行确定性**：所有工具操作失败时必须有降级预案，所有文件引用必须可跳转。
6. **闭环确定性**：所有内建流程与 Skill 都必须具备预检、验收、沉淀和失败处理。

## 核心收益

- 降低规则和目录漂移
- 降低技能名与触发词不一致
- 降低会话类流程的误调度
- 提升新用户初始化后的可理解性和可维护性
- 避免“只有触发词，没有闭环”的伪能力
