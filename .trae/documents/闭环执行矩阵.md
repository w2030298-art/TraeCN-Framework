# 闭环执行矩阵

## 目标

把 TraeCN-Framework 从“能触发”推进到“能稳定落地、能验收、能回滚、能沉淀”。

这份文档定义所有核心流程的统一执行合同，避免出现：

- 只有触发词，没有实际步骤
- 有步骤，没有验收条件
- 有修改，没有沉淀出口
- 失败后没有降级或回滚

## 通用执行合同

所有内建指令与 Skill 默认遵循以下五段式闭环：

| 阶段 | 必须动作 | 验证方式 | 沉淀动作 |
| --- | --- | --- | --- |
| 预检 | 读取 `.trae/config.md`、关联记忆和相关文件 | `Read` / `LS` / `SearchCodebase` | 复用 `knowledge/` 中已有经验 |
| 执行 | 实际使用工具完成任务，不只输出说明 | 文件变更、命令结果、工具返回值 | 记录关键决策 |
| 诊断 | 代码或文档改动后执行 `GetDiagnostics`，命令失败时分析原始报错 | `GetDiagnostics` / `CheckCommandStatus` | 记录错误与修复路径 |
| 验证 | 重新执行关键命令或读取目标文件确认结果 | `RunCommand` / `Read` / `OpenPreview` | 标记已验证范围与残留风险 |
| 沉淀 | 更新 Memory / Evolve / 文档中的相应位置 | 读取已写入内容进行抽查 | 写入下一步建议 |

## 生命周期指令

### `/初始化`

#### 触发场景
- 首次接入框架
- 缺失 `Core/记忆/` 目录或关键记忆文件
- 用户希望把当前工作区接入 TraeCN 工作流
- 工作区缺少 `.trae/` 框架文件，需要自动部署

#### 标准步骤

##### 阶段 A：`.trae/` 部署（当工作区不存在 `.trae/` 时）
1. **解析参数**：判断模式（无参数 / URL / @索引）。
2. **获取配置**：
   - URL 模式 → 下载并解压，定位 `.trae/`
   - 索引模式 → 读取 `$HOME/.trae-configs/index.md`，定位配置
   - 无参数模式 → 查找本地 `@default`，不存在则从 GitHub 下载
3. **完整性校验**：检查必须文件（`.trae/config.md`、`.trae/rules/`）和建议文件（`.trae/skills/`、`.trae/documents/`）。
4. **部署**：将 `.trae/` 复制到工作区根目录，清理临时文件。

##### 阶段 B：标准初始化（`.trae/` 就绪后）
5. 读取 `.trae/config.md`、`user_scope.md`、`project_scope.md`、`session-handoff.md`，判断是否为新环境或断点恢复。
6. 检查 `Core/记忆/`、`knowledge/`、`Evolve/`、`Backup/`、`Skills/` 是否存在，缺失则创建。
7. 为缺失的记忆文件写入标准表头，不覆盖已有用户内容。
8. 若存在 `session-handoff.md` 且有未完成事项，主动询问是否继续。
9. 输出当前工作模式：初始化方式、已加载记忆、已启用的开关、建议的下一条指令。

#### 最低验收
- `.trae/config.md` 存在且可读
- `.trae/rules/00-核心身份.md` 存在且可读
- `.trae/rules/` 包含全部 7 个规则文件
- `Core/记忆/user_scope.md` 存在且可读
- `Core/记忆/project_scope.md` 存在且可读
- `Core/记忆/session-handoff.md` 存在且可读
- `Core/记忆/knowledge/errors.md` 存在且可读
- `Core/记忆/knowledge/decisions.md` 存在且可读
- `Core/记忆/knowledge/patterns.md` 存在且可读

#### 失败处理
- **下载失败**：提示检查网络/URL，建议手动下载或配置本地索引。
- **校验失败（缺少必须文件）**：列出缺失文件，终止并建议检查来源。
- **`.trae/` 已存在冲突**：默认不覆盖，提供覆盖/合并/取消选项。
- **Read 失败**：先用 `Glob`/`LS` 重新定位路径，再补建缺失文件。
- **模板写入失败**：缩小写入范围，只写最小可用表头并重新验证。
- **发现旧会话残留**：不自动清空，先提示用户继续或开始新任务。

#### 回滚策略
- `.trae/` 部署失败 → 清理已复制的部分文件，提示手动部署。
- 标准初始化失败（目录创建失败）→ 检查工作区写入权限，提示以管理员身份运行。

### `/重置`

#### 触发场景
- 用户明确要求清理记忆和会话状态
- 记忆体系严重污染，且用户接受重建

#### 标准步骤
1. 明确说明会被清理的内容和不会动的内容。
2. 获得用户明确确认。
3. 备份 `Core/记忆/` 到 `Backup/reset-时间戳/`。
4. 重建 `user_scope.md`、`project_scope.md`、`session-handoff.md` 与 `knowledge/` 下的标准模板。
5. 读取新文件确认可用，再提示用户重新执行 `/初始化`。

#### 最低验收
- 备份目录实际存在且包含重置前记忆文件。
- 新记忆文件存在且为可读模板。
- `.trae/`、`docs/`、`Builds/`、`Skills/`、`Backup/` 保持不变。

#### 回滚策略
- 任一步骤失败，优先从最新 `Backup/reset-时间戳/` 回滚 `Core/记忆/`。
- 回滚后必须记录失败原因、已恢复范围和未完成项。

### `/结束`

#### 触发场景
- 用户表示“先暂停”“下次继续”“先到这里”“结束会话”，或明确输入 `/结束`
- 上下文过长，需要安全交接

#### 标准步骤
1. 总结本轮目标、已完成内容、未完成内容。
2. 根据用户语义选择结束会话档位：
   - 轻量结束会话：优先更新 `session-handoff.md`，写明当前状态为“待继续”或同等语义，并记录恢复入口、下一步建议和阻塞点。常用于用户说“先暂停”、“下次继续”等场景。
   - 标准结束会话：完整更新 `session-handoff.md`，并将长期有效的信息写入 `project_scope.md`、`user_scope.md`、`knowledge/decisions.md`。常用于用户说“结束会话”或输入 `/结束` 场景。
3. 若有已解决错误或高价值经验，补写 `knowledge/errors.md`、`knowledge/patterns.md`、`Evolve/`。
4. 输出“下次从哪里继续”而不是只说结束。

#### 最低验收
- `session-handoff.md` 已包含本轮状态、已完成/未完成、恢复入口与下一步。
- 轻量结束会话至少保证 `session-handoff.md` 可恢复；标准结束会话至少一个长期记忆文件完成更新，除非本轮无新信息可沉淀。
- 回复中明确给出下一步建议。

#### 失败处理
- 记忆文件写入失败时，先保证 `session-handoff.md` 成功落盘。
- 若系统级故障阻断全部写入，至少保留最小交接摘要。

## Skill 执行合同

### `/分析`
- 产出：目标澄清、约束、任务拆解、复杂度、风险。
- 验收：不是泛泛建议，必须结合当前仓库结构与已有记忆。

### `/设计`
- 产出：方案对比、推荐方案、受影响文件、接口/数据流、落地顺序。
- 验收：必须给出可执行到文件级的实现路径，不能停留在概念层。

### `/视觉设计`
- 产出：需求重述、场景与约束、3 个差异化视觉方向、推荐方向、可执行设计产物（配色/版式/提示词/草图说明）、设计评审报告（基于 5 维度评审模型）、验证范围与风险、下一步建议。
- 方法：默认遵循 7 阶段流程（哲学选型 -> 场景建模 -> 方向发散 -> 方案收束 -> 设计评审 -> 落地执行 -> 最终校验）。
- 模板：优先参考 `.trae/skills/视觉设计/output-template.md` 组织输出；需求较轻时可降级为简化模板。
- 验收：必须包含至少一次基于「哲学、层级、细节、功能、创新」的设计评审，且产出具体的设计参数或执行路径。

### `/测试`
- 产出：复现路径、根因、修复动作、验证结果、残留风险。
- 验收：至少完成一次“复现/诊断/验证”闭环，不能只有静态猜测。

### `/部署`
- 产出：部署目标、配置修改、验证命令、监控/回滚方案。
- 验收：至少完成一次构建或部署前验证，明确环境变量与失败降级。

### `/复盘`
- 产出：本轮成果、问题、决策、模式、后续优化建议。
- 验收：必须同步 Memory 与 `Evolve/`；只有出现真实高复用资产时才考虑写入 `Modules/`。

### `/项目归档`
- 产出：归档目录、版本信息、归档原因、恢复入口、记忆更新。
- 验收：归档路径真实存在，项目状态已从进行中切换为已归档。

## Hooks 闭环

### `pre-edit`
- 读取 `.trae/config.md` 与 `knowledge/patterns.md`
- 目的：避免重复犯错或偏离现有模式

### `on-error`
- 捕获原始错误
- 溯源根因
- 至少尝试两轮不同维度修复
- 修复后重新验证原动作

### `post-build`
- 构建成功后补充执行 `/测试` 逻辑
- 至少记录验证范围与剩余风险

### `pre-push`
- 执行发布前检查或项目要求的额外流程
- 若配置关闭，则显式跳过并说明未执行项

## 本项目当前优先级

### 核心优先
1. 内建生命周期指令产品化
2. Skill 执行深度补齐
3. 自愈、验证、沉淀三个出口打通

### 可选保留
- `Modules/` 仅在出现真实复用资产时启用，不作为当前阶段硬目标
- 模板只保留能减少真实出错率的部分，不再为“看起来完整”而扩张
