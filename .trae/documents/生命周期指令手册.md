# 生命周期指令手册

## 目标

把 `/初始化`、`/重置`、结束会话（系统命令：`/结束`）从“记得怎么做”变成“每次都能按同一标准执行”。

本手册服务于 Coordinator 内建流程，和 `.trae/documents/闭环执行矩阵.md` 配套使用：

- `闭环执行矩阵.md` 负责定义统一合同
- 本文负责定义三个生命周期流程的操作细节

## `/初始化`

### 适用时机
- 新项目首次接入 TraeCN-Framework
- `Core/记忆/` 或 `knowledge/` 缺失
- 用户希望恢复到可开始协作的最小可用状态
- 工作区缺少 `.trae/` 框架文件，需要自动部署

### 参数与分支

```text
/初始化 [参数]
  ├─ .trae/ 已存在 → 直接执行【标准初始化】
  └─ .trae/ 不存在
       ├─ 有 URL 参数 → 【方式一：链接导入】→ 【标准初始化】
       ├─ 有 @索引参数 → 【方式二：本地索引】→ 【标准初始化】
       └─ 无参数 → 【方式三：快捷自动】→ 【标准初始化】
```

### `.trae/` 部署步骤（当工作区不存在 `.trae/` 时）

#### 方式一：链接导入
1. 验证 URL 格式（必须以 `http://` 或 `https://` 开头）
2. 使用 `Invoke-WebRequest` 或 `curl` 下载到临时目录
3. 解压（支持 `.zip` / `.tar.gz`）
4. 在解压内容中定位 `.trae/` 目录
5. 执行【完整性校验】
6. 将 `.trae/` 复制到工作区根目录
7. 清理临时文件

#### 方式二：本地索引
1. 读取 `$HOME/.trae-configs/index.md`
2. 匹配用户指定的索引标识
3. 根据索引路径定位 `.trae/` 目录
4. 执行【完整性校验】
5. 将配置复制到工作区根目录

#### 方式三：快捷自动
1. 检查 `$HOME/.trae-configs/default/.trae/` 是否存在
2. **存在** → 复制到工作区根目录
3. **不存在** → 从 GitHub 官方仓库下载（`https://github.com/qShan1/TraeCN-Framework/archive/refs/heads/main.zip`），解压后提取 `.trae/`，复制到工作区根目录，清理临时文件
4. 执行【完整性校验】

### 完整性校验
部署 `.trae/` 前必须检查（参见 `.trae/configs/default-checksum.md`）：
- **必须存在**：`.trae/config.md`、`.trae/rules/`（至少含 `00-核心身份.md`）
- **建议存在**：`.trae/skills/`、`.trae/documents/`
- 缺少必须文件 → **报错终止**，不继续初始化
- 缺少建议文件 → **警告但继续**，初始化后提示补充

### 冲突处理
若工作区已存在 `.trae/`：
- 默认**不覆盖**，提供覆盖/合并/取消选项
- 必须获得用户明确确认后才能覆盖

### 标准初始化步骤（`.trae/` 就绪后）
1. 读取 `.trae/config.md`、`user_scope.md`、`project_scope.md`、`session-handoff.md`。
2. 检查 `Core/记忆/`、`Core/记忆/knowledge/`、`Evolve/`、`Backup/`、`Skills/` 是否存在。
3. 缺失目录则创建；缺失文件则写入最小模板，不覆盖已有内容。
4. 如果 `session-handoff.md` 存在未完成事项，提示用户是否继续上次任务。
5. 输出初始化摘要：
   - 初始化方式（URL / 本地索引 / 自动下载 / 标准）
   - 已加载的记忆
   - 已创建的目录/文件
   - 当前开关状态
   - 推荐的下一步

### 最小模板要求
- `user_scope.md`：包含标题、`updated`、偏好表头
- `project_scope.md`：包含标题、`updated`、项目状态表头
- `session-handoff.md`：包含标题、`updated`、交接表头
- `knowledge/errors.md`：包含错误记录表头
- `knowledge/decisions.md`：包含关键决策表头
- `knowledge/patterns.md`：包含模式积累表头

### 交付口径
- 不说“初始化完成”之前，必须确认 `.trae/` 结构和关键记忆文件全部可读
- 如果检测到旧会话，不默认忽略，必须把恢复入口讲清楚
- 若 `.trae/` 是通过下载/索引部署的，必须说明部署来源

## `/重置`

### 适用时机
- 用户明确要求清空当前记忆和会话状态
- 当前记忆已严重污染，需要回到干净状态

### 风险说明
- 会清理 `Core/记忆/` 下的用户数据和会话状态
- 不会清理 `.trae/`、`docs/`、`Builds/`、`Skills/`、`Backup/`
- 必须二次确认后才能执行

### 操作步骤
1. 用清单形式说明会清理和保留的范围。
2. 获取用户明确确认。
3. 备份 `Core/记忆/` 到 `Backup/reset-时间戳/`。
4. 重建记忆文件最小模板。
5. 读取回写后的文件，确认可用。
6. 回复中明确：
   - 已保留什么
   - 已清理什么
   - 备份在哪里
   - 如何重新开始

### 失败与回滚
- 备份失败：停止重置，不继续清理
- 模板写入失败：优先回滚 `Core/记忆/`
- 回滚后：写入失败说明和建议下一步

## 结束会话（系统命令：`/结束`）

### 适用时机
- 用户说“先暂停”“先到这里”“下次继续”“结束会话”
- 当前任务阶段性完成，需要安全交接
- 会话上下文过长，需要保存现场

### 操作步骤
1. 汇总本轮目标、已完成项、未完成项。
2. 根据语义选择结束会话档位：
   - **轻量结束会话**：对应“先暂停”、“下次继续”等意图。优先更新 `session-handoff.md`，状态标记为“待继续”，记录恢复入口、下一步建议和当前卡点。
   - **标准结束会话**：对应“结束会话”或输入 `/结束`。在完成 `session-handoff.md` 的基础上（状态标记为“已完成”），再根据本轮新增信息更新 `project_scope.md`、`user_scope.md`、`knowledge/decisions.md`。
3. 如果本轮存在错误修复或新模式，补写 `knowledge/errors.md`、`knowledge/patterns.md`、`Evolve/`。
4. 回复中直接告诉用户“下次从哪里继续”。

### 最小交接要求
- `session-handoff.md` 必须包含：
  - `status`: 待继续 (Light) / 已完成 (Standard)
  - `recovery_entry`: 具体的文件或指令
  - `next_steps`: 建议的 1-3 个动作项
- `project_scope.md`: 更新项目当前进展与 vX.Y.Z 状态。

### 降级策略
- 如果长期记忆写入失败，至少保证 `session-handoff.md` 落盘
- 如果系统异常，保留最小交接摘要并提醒用户下次恢复

## 常见反模式

- 只说“已初始化”，但没有确认记忆文件是否存在
- 只说“已重置”，但没有备份路径
- 只说“下次继续”，但没有恢复入口
- 把旧会话检测结果默默忽略

## 执行检查清单

每次执行生命周期指令前后，至少自检以下问题：

- 是否读取了配置和已有记忆
- 是否明确了创建/清理/保留范围
- 是否验证了关键文件状态
- 是否给出了恢复入口或下一步
- 是否把新信息沉淀回 Memory / Evolve
